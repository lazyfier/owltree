## 目录放置规则（精简版）

### 总原则
**项目级优先，全局级兜底。**

新增代码时，按这个顺序判断：

1. **先放项目级**：优先放进 `projects/<project-name>/`
2. **只有项目内确实找不到合适实现**
3. **且明确需要宿主统一管理**
4. **才允许提升到全局级 `src/`**

### `src/` 放什么
`src/` 只放 **全局级 / 宿主级** 内容：

- 首页与主站页面
- 路由入口
- 全局布局
- 全局主题系统
- 全局 context
- 宿主统一控制的能力
- 模块注册/挂载层

**不要**因为“多个项目可能复用”就自动放进 `src/`。

### `projects/` 放什么
`projects/<name>/` 放 **项目级** 内容：

- 独立项目页面
- 项目私有组件
- 项目私有 hooks
- 项目私有逻辑
- 项目私有样式
- 项目私有资源
- 可整目录替换更新的实现

**默认新增内容都先放项目级。**

### 决策规则
如果一个文件：

- 主要服务某个独立项目
- 未来希望整目录替换
- 不应该影响宿主其他模块

→ 放 `projects/<name>/`

如果一个文件：

- 属于宿主平台本身
- 必须由全局统一控制
- 项目级里没有合适实现

→ 放 `src/`

### 禁止事项
- 不要预先抽“公共组件池”
- 不要因为“以后可能复用”就提前全局化
- 不要让 `projects/a` 直接依赖 `projects/b`
- 不要把项目私有实现混进 `src/`

### 一句话规则
> **默认放项目级；项目内确实找不到，再升级到全局级。**
