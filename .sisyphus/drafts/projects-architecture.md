# Owltree 模块化目录规范

## 目标

把项目拆成两层：

- `src/`：主站 / 宿主平台
- `projects/`：独立项目区

这样以后新增或更新独立项目时，可以尽量只替换 `projects/<project-name>/` 整个目录，而不需要改主站骨架。

---

## 总原则

## 核心优先级

> **项目级优先，全局级兜底。**

决策顺序必须是：

1. 先放项目级：优先在 `projects/<name>/` 内解决
2. 只有项目内确实没有合适实现，且明确需要宿主统一管理时
3. 才提升到全局级，放进 `src/`

这意味着：

- 不预先抽公共
- 不因为“以后可能复用”就提前上提
- 先让每个项目自己闭环
- 全局级只接收真正的平台能力

### `src/` 是平台层

放：

- 首页
- 全站路由
- 导航与布局
- 主题系统
- 全局级组件
- 公共 hooks
- 公共工具函数
- 全局 context
- 模块注册与适配层

一句话：

> 凡是全局级、宿主站依赖、不能随便整体替换的内容，都放 `src/`。

补充规则：

> 进入 `src/` 不是默认选项，而是升级选项。
> 只有项目级无法满足时，才允许升级到全局级。

### `projects/` 是独立项目层

放：

- 独立小游戏
- 独立工具
- 独立实验模块
- 可以整体替换的一块业务

一句话：

> 凡是某个项目私有、未来希望整目录替换更新的内容，都放 `projects/`。

---

## 推荐目录结构

```txt
owltree/
├── src/
│   ├── pages/          # 主站页面
│   ├── components/     # 全站公共组件
│   ├── hooks/          # 公共 hooks
│   ├── contexts/       # 全局 context
│   ├── lib/            # 公共工具函数
│   ├── modules/        # projects 的注册 / 适配层
│   └── App.tsx         # 主路由入口
│
├── projects/
│   ├── moon-throw/
│   ├── project-a/
│   └── project-b/
│
├── docs/
├── theme/
├── archive/
└── .sisyphus/
```

---

## `src/` 应该放什么

### 1. 主站页面

例如：

- 首页
- Games 列表页
- Notes 列表页
- Tools 列表页
- Trends 列表页

这些页面属于 Owltree 本体，不属于某个独立项目。

### 2. 全局级组件

例如：

- 宿主站首页所需组件
- 全局导航组件
- 全局布局组件
- 全局主题切换组件
- 只属于平台壳层的基础组件

规则：

> 不再因为“两个及以上项目复用”就自动进入一个公共组件池。
> 组件只分两层：全局级、项目级。
> 只有明确属于平台壳层的组件，才放 `src/`。

### 3. 公共 hooks / contexts / lib

例如：

- `useTheme`
- `ThemeContext`
- 通用格式化函数
- 公共路由辅助函数

这些是宿主基础设施，不应被独立项目私有化。

### 4. 模块接入层

建议在 `src/modules/` 放宿主对 `projects/` 的注册逻辑。

例如：

- 模块清单
- 路由映射
- 模块元信息读取
- 宿主统一入口

这层的职责不是写业务，而是把 `projects/` 接入主站。

---

## `projects/` 应该放什么

### 1. 独立项目代码

例如：

- `moon-throw` 的组件
- `moon-throw` 的私有游戏规则
- 某个工具的专属表单与逻辑
- 某个实验项目的私有资源

规则：

> 默认组件归项目自己管理。
> 即使多个项目都用了相似组件，也不要先抽到“公共组件层”；
> 先允许各项目各自维护，只有明确上升为平台能力时，才进入 `src/`。

更严格一点：

> 优先在项目内找、项目内写、项目内改。
> 只有项目内确实找不到可用实现，才去考虑全局层。

### 2. 独立项目自己的资源

例如：

- assets
- 项目私有样式
- 项目私有类型
- 项目 README

### 3. 独立项目自己的入口

每个项目都要有固定入口，避免宿主直接引用散乱内部文件。

---

## 单个项目推荐结构

以 `moon-throw` 为例：

```txt
projects/
└── moon-throw/
    ├── module.json
    ├── README.md
    ├── src/
    │   ├── index.ts
    │   ├── pages/
    │   ├── components/
    │   ├── hooks/
    │   ├── engine/
    │   ├── data/
    │   └── styles/
    └── assets/
```

### 推荐约定

- `src/index.ts`
  - 项目唯一公开入口
- `module.json`
  - 项目元信息
- `README.md`
  - 项目说明

---

## `module.json` 建议格式

```json
{
  "id": "moon-throw",
  "name": "月抛模拟器",
  "route": "/moon-throw",
  "entry": "./src/index.ts",
  "version": "1.0.0"
}
```

用途：

- 让宿主知道项目是谁
- 知道挂在哪个路由
- 知道入口文件在哪
- 以后支持版本管理

## 更新策略

你的目标是：

> 以后我拿到一个独立项目文件夹，直接放进来覆盖更新。

这个目标可以成立，但前提是遵守下面规则：

### 允许整目录覆盖的内容

- `projects/<project-name>/` 内部私有实现
- 项目私有组件
- 项目私有样式
- 项目私有资源
- 项目私有逻辑

### 不允许通过覆盖独立项目来修改的内容

- `src/App.tsx`
- 全局主题系统
- 公共组件
- 公共 hooks
- 宿主路由骨架
- 构建配置
- 全局 CSS 变量体系

结论：

> 可以覆盖项目目录，但不能指望覆盖项目目录顺便改宿主平台。

---

## 边界规则

### 规则 1：默认不设“公共组件池”

不要因为“两个及以上项目会用到”就自动抽成公共组件。

先问两个问题：

1. 它是不是平台壳层能力？
2. 它是不是必须被宿主统一控制？

只有答案是“是”，才进入 `src/`。

否则继续留在各自项目里。

并且在做这个判断前，必须先满足一个前提：

> 项目级已经确认没有合适实现。

### 规则 2：项目内容默认留在项目内

如果某段代码主要服务某个独立项目，就放回它自己的 `projects/<name>/`。

即使另一个项目未来可能抄一份，也不要提前抽象。

### 规则 3：项目之间不能互相耦合

禁止这种写法：

```ts
projects/a/... -> import '../../b/src/...'
```

原因：

- 一旦这样做，就失去独立可替换能力
- 更新 A 时会偷偷依赖 B

### 规则 4：宿主只认项目入口，不认内部实现

宿主应该只通过：

- `module.json`
- `src/index.ts`

来接入模块，而不是深入项目内部随处 import。

## 判断一个文件该放哪

### 放 `src/`

满足任一条件：

- 项目级确认没有合适实现
- 宿主页面要用
- 属于平台壳层
- 是全局主题 / 布局 / 路由的一部分
- 必须由宿主统一控制

### 放 `projects/<name>/`

满足全部条件更合适：

- 主要服务某个独立项目
- 未来希望整目录替换
- 不应该影响宿主其他模块
- 业务含义明显只属于该项目
- 即使别的项目也可能出现类似实现

默认新增内容，应先按这一类处理。

---

## 适合迁入 `projects/` 的内容

当前 Owltree 里，未来如果做模块化，最适合迁出的通常是：

- `moon-throw` 整体游戏 UI
- `moon-throw` 游戏引擎
- `moon-throw` 游戏数据

也就是把它逐步收成一个完整独立项目。

而这些继续留在 `src/` 更合理：

- 首页 / Games 页 / Tools 页这些主站页面
- ThemeContext
- 全局主题变量与公共布局
- 宿主站壳层组件

---

## 最终建议

把 Owltree 视为：

- `src/` = 平台
- `projects/` = 独立业务块

以后新增模块时：

1. 先判断它是不是独立项目
2. 如果是，就新建 `projects/<name>/`
3. 如果有复用价值，再把公共部分回提到 `src/`

不要反过来：

- 不要先把所有东西都放 `projects/`
- 也不要把独立项目继续塞进 `src/` 深处

---

## 一句话结论

可以按你说的做：

- `src` 放主页和宿主平台
- `projects` 放独立项目

但补一条最重要的规则：

> 不按“公共组件”来管理，而是按“全局级 / 项目级”来管理。
> 默认归项目，优先项目级；项目内确实找不到，再升级到全局级 `src`。
